rzenc no-deps Rust Windows & Linux

Simple, careful file encryption CLI that replaces the target file atomically. Uses ChaCha20 + HMAC‑SHA‑256 with PBKDF2 (no external crates).

Quickstart Features Safety & limits On‑disk format Algorithms Platform notes Troubleshooting

Quickstart

Build

# rustc 1.70+ recommended (stable)
rustc -O rzenc.rs

No third‑party crates required; standard library + a few FFI calls only.

Encrypt

# recommended: set the passphrase via env var
set RZENC_PASS=correct horse battery staple   # Windows (cmd)
# export RZENC_PASS='correct horse battery staple'       # Linux/macOS

rzenc E a.txt   # replaces a.txt atomically with an encrypted version

Decrypt

rzenc D a.txt   # verifies tag, then replaces a.txt with plaintext

If RZENC_PASS is not set, the tool reads the passphrase from stdin (echoed).

Features at a glance

  • Encrypt‑then‑MAC: ChaCha20 stream cipher + HMAC‑SHA‑256 for integrity.
  • Password‑based keys: PBKDF2‑HMAC‑SHA‑256 with per‑file 16‑byte salt and 600,000 iterations.
  • Data safety first: streams to a temp file, fsyncs, closes handles, then atomically replaces the original (Windows: ReplaceFileW, Unix: rename).
  • Verify before decrypt: the MAC is checked before any plaintext is written; wrong passwords never corrupt your file.
  • No dependencies: uses OS CSPRNG (Windows RtlGenRandom, Unix /dev/urandom).

Safety model & limitations

⚠️ Important: Hand‑rolled crypto is risky compared to audited libraries. This project is for environments where dependencies are impossible. For high‑stakes data, prefer audited crates like chacha20poly1305, argon2, and getrandom.

  • Atomic replacement, not byte‑in‑place: avoids corruption on crash; you still invoke it as E <file>/D <file> and the same path is replaced.
  • Authentication first: decryption refuses to write if the tag fails (bad password/tamper).
  • No secure wipe: replacing a file does not erase old sectors; journaled filesystems or SSDs may retain copies.
  • Password input: stdin is echoed to remain no‑deps/portable. Use RZENC_PASS or patch in a no‑echo prompt via OS FFI if required.
  • Memory wiping is best‑effort: key material is scrubbed with volatile writes, but full OS/optimizer guarantees are not absolute.
  • Compatibility: File format is versioned via MAGIC "RZENC001". Future versions may evolve.

[ CIPHERTEXT ... ]
[ TAG(32) ]

Algorithms & parameters

  • Key derivation: PBKDF2‑HMAC‑SHA‑256(password, salt, iterations=600,000) → 64 bytes.
    • First 32 bytes → ChaCha20 key.
    • Next 32 bytes → HMAC‑SHA‑256 key.
  • Nonce: 12 random bytes; ChaCha20 block counter starts at 1.
  • MAC: HMAC‑SHA‑256(header || ciphertext); verified before decrypting.
  • Randomness: Windows RtlGenRandom via FFI; Unix reads /dev/urandom.

Platform notes

  • Windows: Atomic replacement uses ReplaceFileW with write‑through. Handles are closed before replacement. Linkage to advapi32 (RNG) and kernel32 is declared in the source.
  • Linux/Unix: After rename(), the parent directory is fsynced (best‑effort) to harden against power loss.
  • Large files: Processed in 1 MiB chunks; only modest RAM usage.

Command‑line usage

Usage: rzenc [E|D] <file>

Environment:
  RZENC_PASS   passphrase (recommended way)

Examples:
  # Encrypt
  set RZENC_PASS=top secret              && rzenc E a.txt      # Windows (cmd)
  # export RZENC_PASS='top secret'       && rzenc E a.txt      # Linux/macOS

  # Decrypt
  set RZENC_PASS=top secret              && rzenc D a.txt

  # Verify integrity behavior
  copy a.txt a.enc.bak     # or cp a.txt a.enc.bak
  rzenc E a.txt
  # flip a byte in the encrypted file, then:
  rzenc D a.txt            # -> "authentication failed" and original is left untouched

Exit codes: 0 success, 1 runtime error, 2 usage error.

Troubleshooting

  • Link errors on Windows: ensure you’re using the MSVC or MinGW toolchain with standard Windows SDK libs; the source declares #[link(name="advapi32")] and #[link(name="kernel32")].
  • "Access is denied" on replace: make sure the target isn’t open in another program; the tool closes its own handles before replacing.
  • Slow PBKDF2: 600k iterations is intentional. You can change DEFAULT_PBKDF2_ITERS in the source to trade speed vs. brute‑force cost.
  • Passphrase echo: set RZENC_PASS to avoid echoed input; or add a no‑echo OS prompt if you’re comfortable extending the source.

Hardening ideas (if you allow minimal deps)

  • Swap PBKDF2 for a memory‑hard KDF (argon2 / scrypt).
  • Use an AEAD like chacha20poly1305 to reduce moving parts.
  • No‑echo password input (rpassword), and portable RNG (getrandom).
